# 服务器 2.1 全量 16000 步训练说明

这份说明对应 `formal_v2` 的长时程版本：

- 架构空间仍使用 `L8`
- block 类型仍为 `M / A`
- 全量合法架构共 `256` 个
- 每个架构从头训练 `16000` 步
- 训练完成后聚合成新的 `meta_dataset.csv`

这条线只 scale 训练步数，不改变模型代码和架构空间。建议把它视为 `formal_v2_full256_steps2000` 的长时程对应数据集，不要和旧的 `1000` 步或 `2000` 步 family 目录混写。

## 1. 本地 2000 步结果估时

本地已经拉回了 2000 步正式结果：

- `C:\tmp\meta-model-train-formal-v2-full256-2000\formal_v2_full256_steps2000_a_family_summary.csv`
- `C:\tmp\meta-model-train-formal-v2-full256-2000\formal_v2_full256_steps2000_b_family_summary.csv`

根据两个 `family_summary.csv` 的 `runtime_s` 字段：

- 总架构数：`256`
- shard A：`128` 个架构，总运行约 `5227s = 1.45h`
- shard B：`128` 个架构，总运行约 `4505s = 1.25h`
- 单架构 2000 步平均约 `38.0s`
- 单架构 2000 步中位数约 `37.8s`
- 单架构 2000 步 p90 约 `48.3s`
- 单架构 2000 步最大约 `61.2s`

双 shard 并行时，2000 步 wall time 主要由 shard A 决定，约 `1.45h`。因此 16000 步粗略估计为：

```text
1.45h * 8 = 11.6h
```

实际运行可能因为长时间 GPU 状态、验证开销和系统负载略有浮动。建议预留 `12-14h`。

## 2. 重要限制

当前训练脚本只支持：

- 跳过已经完整完成的 architecture
- 对未完成的 architecture 重新从头训练

当前不支持：

- 把已经完成的 `2000` 步 architecture 无缝继续训练到 `16000` 步

原因是 family 训练流程不保存模型 checkpoint，只保存：

- `metrics.csv`
- `summary.json`
- `family_summary.csv`

因此 `16000` 步必须使用新的 family 名称，从头跑完整目标数据集。

## 3. 推荐使用的一键脚本

新增脚本：

- [scripts/run_formal_v2_full256_steps16000_two_shards_and_merge.sh](C:/Users/11599/Desktop/本科生科研/毕业论文/multiple_physics_pretraining-main/meta-model-train/scripts/run_formal_v2_full256_steps16000_two_shards_and_merge.sh)

它会做这些事：

- 如果缺少 `formal_v2_256` 架构 CSV，就自动生成
- 自动把 `256` 个架构分成两个 shard：`0-127`、`128-255`
- 两个 shard 并行训练
- 训练结束后自动执行 `build_meta_dataset.py`
- 打印两个 shard 的 `family_summary.csv` 行数和最终 `meta_dataset.csv` 行数

默认输出命名：

- shard A: `formal_v2_full256_steps16000_a`
- shard B: `formal_v2_full256_steps16000_b`
- merged csv: `formal_v2_full256_steps16000_meta_dataset.csv`

## 4. 服务器启动步骤

先在服务器同步仓库：

```bash
cd /root/meta-model-train
git pull
source /root/miniconda3/etc/profile.d/conda.sh
conda activate mpp-meta
chmod +x scripts/server_launch_family_shard.sh
chmod +x scripts/run_formal_v2_full256_steps16000_two_shards_and_merge.sh
```

然后建议直接用 `tmux` 启动双分片任务：

```bash
tmux new -s formal_v2_full256_steps16000
cd /root/meta-model-train
bash scripts/run_formal_v2_full256_steps16000_two_shards_and_merge.sh
```

退出 `tmux` 但不终止任务：

```bash
Ctrl+b 然后按 d
```

重新进入：

```bash
tmux attach -t formal_v2_full256_steps16000
```

查看日志：

```bash
cd /root/meta-model-train
ls logs/formal_v2_full256_steps16000
tail -n 20 logs/formal_v2_full256_steps16000/shard_a.log
tail -n 20 logs/formal_v2_full256_steps16000/shard_b.log
```

## 5. 运行中估算剩余时间

可以用日志中的当前 architecture 进度粗略判断：

```bash
grep "start " logs/formal_v2_full256_steps16000/shard_a.log | tail
grep "start " logs/formal_v2_full256_steps16000/shard_b.log | tail
```

也可以看某个正在跑的架构的 step 和 elapsed：

```bash
tail -n 50 logs/formal_v2_full256_steps16000/shard_a.log
tail -n 50 logs/formal_v2_full256_steps16000/shard_b.log
```

如果某个 shard 因临时问题中断，修复后重新运行同一个脚本即可：

- 已完整完成且有 `summary.json` 的 architecture 会被 `--skip_existing` 跳过
- 未完成的 architecture 会从头重跑
- 不会把 `2000` 步旧 family 混入 `16000` 步新 family

## 6. 完成后的最小验证

跑完后检查：

```bash
cd /root/meta-model-train
wc -l outputs/toy_diffusion/meta_model_family/formal_v2_full256_steps16000_a/family_summary.csv
wc -l outputs/toy_diffusion/meta_model_family/formal_v2_full256_steps16000_b/family_summary.csv
wc -l outputs/toy_diffusion/meta_model_family/formal_v2_full256_steps16000_meta_dataset.csv
head -n 5 outputs/toy_diffusion/meta_model_family/formal_v2_full256_steps16000_meta_dataset.csv
tail -n 5 outputs/toy_diffusion/meta_model_family/formal_v2_full256_steps16000_meta_dataset.csv
```

默认配置下：

- 每个 shard 的 `family_summary.csv` 应该是 `129` 行
  - `128` 个架构
  - `1` 行表头
- 聚合后的 `meta_dataset.csv` 应该是 `82433` 行
  - 每个架构 `322` 行
  - 共 `256 * 322 = 82432` 行数据
  - 再加 `1` 行表头

其中每个架构的 `322` 行来自：

- `1` 行初始化验证：`step = -1`
- `321` 行周期验证

在默认 `val_every=50`、`max_steps=16000` 下，验证 step 为：

- `0, 50, 100, ..., 15950, 15999`
