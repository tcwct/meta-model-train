# 服务器 2.0 全量 2000 步训练说明

这份说明对应当前 `2.0` 主线：

- 架构空间使用 `L8`
- 全量合法架构共 `256` 个
- 服务器训练目标改为每个架构 `2000` 步
- 训练完成后聚合成新的 `meta_dataset.csv`

建议把这条线视为一套新的正式数据集，不要和旧的 `full328` 或旧的 `1000` 步 family 混用。

## 1. 先说明一个重要限制

当前训练脚本只支持：

- 跳过已经完整完成的 architecture
- 对未完成的 architecture 重新从头训练

当前**不支持**：

- 把已经完成的 `1000` 步 architecture 无缝继续训练到 `2000` 步

原因是现在不会保存模型 checkpoint，只保存：

- `metrics.csv`
- `summary.json`
- `family_summary.csv`

因此如果你之前服务器上的 2.0 部分结果是按 `1000` 步跑完的，那么这批结果**不能直接续到 2000 步**。正确做法是：

- 新开一个新的 `family_name`
- 用 `2000` 步重新跑完整目标数据集

仓库现在已经加入保护逻辑：如果你误用了旧 family 目录，且训练配置和已有 `family_manifest.json` 不一致，脚本会直接报错退出，而不是悄悄混写。

## 2. 推荐使用的一键脚本

仓库里新增了一个专门面向这条主线的脚本：

- [scripts/run_formal_v2_full256_steps2000_two_shards_and_merge.sh](C:/Users/11599/Desktop/本科生科研/毕业论文/multiple_physics_pretraining-main/meta-model-train/scripts/run_formal_v2_full256_steps2000_two_shards_and_merge.sh)

它会做这些事：

- 如果缺少 `formal_v2_256` 架构 CSV，就自动生成
- 自动把 `256` 个架构分成两个 shard：`0-127`、`128-255`
- 两个 shard 并行训练
- 训练结束后自动执行 `build_meta_dataset.py`
- 打印两个 shard 的 `family_summary.csv` 行数和最终 `meta_dataset.csv` 行数

默认输出命名：

- shard A: `formal_v2_full256_steps2000_a`
- shard B: `formal_v2_full256_steps2000_b`
- merged csv: `formal_v2_full256_steps2000_meta_dataset.csv`

## 3. 服务器启动步骤

先在服务器同步仓库：

```bash
cd /root/meta-model-train
git pull
source /root/miniconda3/etc/profile.d/conda.sh
conda activate mpp-meta
chmod +x scripts/server_launch_family_shard.sh
chmod +x scripts/run_formal_v2_full256_steps2000_two_shards_and_merge.sh
```

然后建议直接用 `tmux` 启动一整个双分片任务：

```bash
tmux new -s formal_v2_full256
cd /root/meta-model-train
bash scripts/run_formal_v2_full256_steps2000_two_shards_and_merge.sh
```

退出 `tmux` 但不终止任务：

```bash
Ctrl+b 然后按 d
```

重新进入：

```bash
tmux attach -t formal_v2_full256
```

查看日志目录：

```bash
cd /root/meta-model-train
ls logs/formal_v2_full256_steps2000
tail -n 20 logs/formal_v2_full256_steps2000/shard_a.log
tail -n 20 logs/formal_v2_full256_steps2000/shard_b.log
```

## 4. 如果你想沿用“之前已经开始但尚未完成”的 2000 步结果

只有在下面两个条件同时满足时，才建议复用已有 family 目录：

- 之前那批结果本来就是按这套 `256` 架构、`2000` 步配置启动的
- family 名称与 shard 划分保持一致

在这种情况下，重新运行同一个脚本即可：

- 已完成的 architecture 会被 `--skip_existing` 跳过
- 未完成的 architecture 会重新从头训练

如果你之前的结果是：

- 只跑了部分架构，但每个已完成架构只有 `1000` 步
- 或者使用了不同的 family 名称 / shard 划分 / 架构 CSV

那就不要复用旧目录，直接启一套新的 `formal_v2_full256_steps2000*` 输出。

## 5. 最小验证命令

跑完后建议先检查：

```bash
cd /root/meta-model-train
wc -l outputs/toy_diffusion/meta_model_family/formal_v2_full256_steps2000_a/family_summary.csv
wc -l outputs/toy_diffusion/meta_model_family/formal_v2_full256_steps2000_b/family_summary.csv
wc -l outputs/toy_diffusion/meta_model_family/formal_v2_full256_steps2000_meta_dataset.csv
head -n 5 outputs/toy_diffusion/meta_model_family/formal_v2_full256_steps2000_meta_dataset.csv
tail -n 5 outputs/toy_diffusion/meta_model_family/formal_v2_full256_steps2000_meta_dataset.csv
```

如果使用默认配置：

- 每个 shard 的 `family_summary.csv` 应该是 `129` 行
  - `128` 个架构
  - `1` 行表头
- 聚合后的 `meta_dataset.csv` 应该是 `10753` 行
  - 每个架构 `42` 行
  - 共 `256 * 42 = 10752` 行数据
  - 再加 `1` 行表头

其中每个架构的 `42` 行来自：

- `1` 行初始化验证：`step = -1`
- `41` 行周期验证

在默认 `val_every=50`、`max_steps=2000` 下，验证 step 为：

- `0, 50, 100, ..., 1950, 1999`
